# STRATEGIC MATURITY MODEL

## OVERVIEW
Focusing on DTC/E-commerce business generating up to £10m in revenue I have developed a process using the Google Ecoysystem to better understand LTV which is supported by a robust set of metrics. it takes DTC/E-commerce brands from raw data to  **Strategic Maturity**. 

## 🚀 The Core Framework
The repository is organized into four distinct modules:

| Module | Scope | Description |
| :--- | :--- | :--- |
| **1.0** | **Validation** | Data integrity guardrails, ensuring financial and data accuracy. |
| **2.0** | **Segmentation** | Analyzing the customer base by purchase frequency and profit buckets. |
| **3.0** | **Migration** | Tracking matured LTV and side-by-side performance of yearly cohorts. |
| **4.0** | **Strategic Bridge** | Deep dives into CAC payback, bridge financials, and unit economic milestones. |

---

## 🛠️ The "Minefield" Testing Suite
To ensure the infrastructure is robust, this repo includes two testing datasets.

### 🔴 The Red Dataset (Stress Test)
* **Purpose:** To intentionally trigger errors and validate the "Security" scripts.
* **Key Tests:** Case-sensitivity splits (`c001` vs `C001`), division-by-zero math errors, and future-dated orders.
* *Success = Validation scripts (1.x) flagging these rows for exclusion.*

### 🟢 The Green Dataset (Migration Flow)
* **Purpose:** To demonstrate a perfect 90-day LTV migration.
* **Key Tests:** Correct calculation of "Bridge" rates and sequential order counts over a 3-month window.

## 📂 Comprehensive File Index (The "Cheat Sheet")

### **1.0 Data Integrity & Validation**
* **1.1 Primary Key Audit:** Detects if the same Order ID is assigned to multiple customers or dates.
* **1.2 Identical Transactions:** Identifies "Ghost Duplicates" where the exact same transaction occurs twice.
* **1.3 Identity Cleaning:** Normalizes `customer_id` strings (handling casing and blank spaces).
* **1.4 Date Validation:** Flags "Future Orders" and identifies potential DD/MM vs MM/DD format flips.
* **1.5 Financial Guardrails:** Uses `SAFE_DIVIDE` to prevent system crashes on zero-revenue test orders.

### **2.0 Performance & Segmentation**
* **2.1 Cohort Volume:** Tracks the total number of customers acquired in specific time periods.
* **2.2 Revenue Concentration:** Identifies the top 10% of customers driving the majority of profit.
* **2.3 Frequency Buckets:** Groups customers by "number of times purchased" (1x, 2x, 3x+).
* **2.4 Profit Per Segment:** Breaks down the actual net profit generated by each frequency bucket.

### **3.0 LTV & Cohort Maturation**
* **3.1 Raw LTV:** The "as-it-is" view of total spend divided by total customers.
* **3.2 Matured LTV:** Calculates the realized value of a cohort after it has "aged" 12+ months.
* **3.3 Cohort Side-by-Side:** Compares the quality of 2023 customers vs 2024 customers at the same age.
* **3.4 Days Between Orders:** Calculates the "Retention Velocity" to inform email marketing timing.

### **4.0 The Strategic Bridge (Consultant View)**
* **4.1.3 The Bridge Financials:** Calculates the specific profit jump achieved when a customer makes purchase #2.
* **4.1.5 Granular Bridge Audit:** A deep-dive into which specific products are the best "Bridge-Builders."
* **4.2 Retention Yield:** Measures the long-term ROI of retention efforts vs acquisition spend.
* **4.3 CAC Payback Milestone:** The definitive view on exactly which day a customer becomes "Net Profitable."

## 📈 Implementation Guide
1. **Ingestion:** Upload `red_dataset.csv`.
2. **Audit:** Run scripts **1.1 through 1.5**. Ensure the "Minefield" rows are successfully caught.
3. **Visualization:** Switch to `green_dataset.csv` and run **4.1.3** to see the Bridge Financials in action.
4. **Live Connection:** Update the `FROM` clauses to point to your production BigQuery events table.
